# DungeonRush 代码总览

## 1. 项目定位

DungeonRush 是一个使用 C 与 SDL 构建的 2D 动作游戏。玩法核心是“贪吃蛇式队伍增长 + 地牢战斗 + 随机地图 + 武器/Buff/掉落系统”。

从代码组织看，项目不是按面向对象层级划分，而是按“功能模块 + 全局状态”划分：

- `main.c` 负责程序入口。
- `res.c` 负责 SDL、窗口、字体、图片、音频等资源初始化。
- `ui.c` 负责主菜单、选项界面、排行榜界面、联机入口。
- `game.c` 负责游戏主状态、初始化、主循环、碰撞、胜负、掉落、阶段推进。
- `map.c` 负责地图生成与地图块渲染数据准备。
- `render.c` 负责统一动画列表、绘制顺序、UI 绘制、血条和屏幕过渡效果。
- `player.c` / `sprite.c` / `weapon.c` / `bullet.c` 负责实体层数据与战斗行为。
- `ai.c` 负责怪物 AI。
- `audio.c` 负责背景音乐与音效播放。
- `net.c` 负责局域网联机同步。
- `storage.c` 负责本地排行榜持久化。

整体设计偏“过程式状态机”：大量核心状态定义在全局变量中，由 `gameLoop()` 在每帧中按固定顺序推进。

## 2. 目录与模块划分

### 2.1 入口与资源生命周期

执行入口在 `src/main.c`：

1. `prngSrand(time(NULL))` 初始化随机数种子。
2. `init()` 初始化 SDL 子系统、窗口、渲染器、字体、音频、网络。
3. `loadMedia()` 加载贴图、文本、武器模板、公共角色模板、音频资源。
4. `mainUi()` 进入主菜单。
5. 退出时由 `cleanup()` 释放窗口、渲染器、纹理、音频对象并关闭 SDL。

这说明整个程序采用“全局一次初始化，多局游戏复用资源”的模型：资源在程序启动时加载一次，而不是每局重新加载。

### 2.2 UI 模块

`src/ui.c` 管理所有非战斗界面：

- `baseUi()`：初始化一个简单背景地图，作为 UI 背景。
- `chooseOptions()`：通用菜单选择逻辑，使用一个临时 `Snake` + `Sprite` 作为菜单光标。
- `mainUi()`：主菜单，支持单人、双人/局域网、排行榜、退出。
- `chooseLevelUi()`：选择关卡等级。
- `launchLocalGame()`：启动本地游戏，结束后展示分数并写入排行榜。
- `launchLanGame()`：主机/加入联机房间。
- `rankListUi()` / `localRankListUi()`：展示排行榜。

UI 模块本质上是游戏外层状态机，负责把用户从菜单流转到 `startGame()`。

### 2.3 游戏核心模块

`src/game.c` 是项目的中心控制器，负责：

- 每局/每阶段初始化：`initGame()`
- 多阶段串联：`startGame()`
- 每帧驱动：`gameLoop()`
- 游戏结束清理：`destroyGame()`
- 玩家、怪物、道具、Buff、掉落、伤害、碰撞、阶段结算

这个文件同时保存了大量游戏全局变量，例如：

- 地图：`map[][]`、`itemMap[][]`、`hasMap[][]`、`hasEnemy[][]`
- 实体：`spriteSnake[]`、`bullets`
- 计数器：`spritesCount`、`playersCount`、`flasksCount`、`herosCount`
- 难度参数：`GAME_LUCKY`、`GAME_DROPOUT_WEAPONS`、`GAME_TRAP_RATE`、`GAME_MONSTERS_HP_ADJUST` 等
- 阶段与终局状态：`gameLevel`、`stage`、`status`、`willTerm`、`termCount`

可以把 `game.c` 理解为一个“调度中心 + 规则中心”。

### 2.4 地图模块

`src/map.c` 负责随机地图生成与地图块实例化：

- `initPrimMap()`：先在半尺寸网格上生成原始地图。
- `cellularAutomata()`：用元胞自动机平滑洞穴结构。
- `initRandomMap()`：把粗网格扩展到完整网格，放置陷阱与出口。
- `initMap()` / `initBlock()`：把逻辑地图转成 `Block`。
- `decorateMap()`：给地板加入装饰变化。
- `pushMapToRender()`：根据地形邻接关系，把墙、地面、前景墙等动画推入渲染列表。

因此，地图模块分为两层：

- 逻辑层：`hasMap`、`isTrap`、`primMap`
- 表现层：`map[][]` 中的 `Block` 与渲染动画

### 2.5 渲染模块

`src/render.c` 提供统一的渲染抽象。核心不是直接渲染实体，而是渲染 `Animation`：

- 所有内容都通过 `animationsList[ANIMATION_LINK_LIST_NUM]` 分类存放。
- 分类包括地图、特效、道具、死亡效果、角色、UI、前景墙等。
- `createAndPushAnimation()` 用于创建并压入某个渲染链表。
- `updateAnimationLinkList()` 更新动画帧和生命周期。
- `renderAnimation()` 真正把一帧动画绘制到屏幕。
- `render()` 负责按顺序渲染各层内容。

这个设计让地图、角色、特效、UI 都统一到一套“动画对象”上，简化了表现层组织。

### 2.6 实体与战斗模块

相关文件包括：

- `player.c/h`：`Snake` 创建与销毁
- `sprite.c/h`：`Sprite` 创建、方向、位置缓冲
- `weapon.c/h`：武器模板与攻击参数
- `bullet.c/h`：投射物行为

游戏把一个角色队伍抽象成 `Snake`，而不是单独 Player 对象。一个 `Snake` 内部用链表保存多个 `Sprite`，表现出“蛇身/队伍”的增长效果。

### 2.7 AI 模块

`src/ai.c/h` 负责怪物决策：

- `AiInput(Snake* snake)` 决定怪物下一步移动方向。
- `getPowerfulPlayer()` 用于判定当前更强的玩家，服务于 UI 或 AI 逻辑。

在 `gameLoop()` 中，AI 不是每帧都算，而是按 `AI_DECIDE_RATE` 节流执行。

### 2.8 音频模块

`src/audio.c/h` 封装背景音乐与音效：

- `playBgm()`、`randomBgm()`、`stopBgm()`
- `playAudio()` 播放事件音效
- `pauseSound()`、`resumeSound()`

音频资源的加载放在 `res.c` 中，播放控制放在 `audio.c` 中，属于“资源加载”和“运行时控制”分离。

### 2.9 联机模块

`src/net.c/h` 负责局域网对战：

- `hostGame()` / `joinGame()` 建立连接
- `sendPlayerMovePacket()` 同步方向输入
- `sendGameOverPacket()` 同步结束状态
- `recvLanPacket()` 接收数据包

协议非常轻量，使用位域压缩到 32 位结构，主要同步：

- 握手包
- 玩家移动包
- 游戏结束包

这说明联机并不是完整状态同步，而更接近“输入同步/事件同步”。

### 2.10 存档与辅助模块

- `storage.c/h`：本地排行榜的读取、写入、插入排序
- `helper.c/h`：随机数、矩形碰撞、距离、包围盒等工具函数
- `types.c/h`：基础类型、动画、文本、分数等通用对象实现
- `adt.h`：双向链表 `LinkList` / `LinkNode`
- `prng.c/h`：伪随机数相关
- `text.h`：若干文本资源索引常量

## 3. 核心数据结构

### 3.1 基础渲染类型

`src/types.h` 定义了程序最基础的表现层对象：

#### `Texture`

表示一组纹理帧：

- `origin`：SDL 纹理对象
- `width` / `height`
- `frames`
- `crops`：每帧在原始纹理中的裁剪矩形

这是游戏中所有动画帧的源数据。

#### `Text`

表示渲染好的文字纹理：

- 文本内容 `text`
- 宽高
- SDL 纹理对象 `origin`
- 颜色

#### `Effect`

表示颜色/透明度变化效果：

- `duration`
- `currentFrame`
- `length`
- `keys`：关键帧颜色数组
- `mode`：混合模式

它不是独立可见对象，而是给 `Animation` 叠加视觉效果。

#### `Animation`

这是渲染系统的核心结构：

- 指向 `Texture`
- 可选 `Effect`
- 循环方式 `LoopType`
- 当前帧、持续时间、位置、旋转、翻转
- `at`：锚点类型
- `bind`：可绑定到某个 `Sprite`
- `dieWithBind`：绑定目标死亡时是否销毁
- `lifeSpan`：寿命型动画的剩余时长

几乎所有可见对象最终都会转成 `Animation` 进入渲染队列。

### 3.2 游戏世界对象

#### `Score`

保存统计信息：

- `damage`：造成伤害
- `stand`：承受伤害
- `killed`：击杀数
- `got`：收集/获得数量
- `rank`：综合评分

#### `Block`

表示地图格子：

- `bp`：地块类型，可能是陷阱、墙、地板、出口
- `x` / `y`
- `bid`：贴图资源编号
- `enable`：是否激活，主要给陷阱/出口用
- `ani`：对应的表现动画

#### `Item`

表示地图上的拾取物：

- `type`：英雄、红药、黄药、武器、空
- `id`：具体资源或武器编号
- `belong`：武器适配的角色类型
- `ani`：显示动画

### 3.3 队伍与角色对象

#### `Snake`

`Snake` 是游戏最重要的逻辑单位，代表一个阵营中的一整条队伍：

- `sprites`：双向链表，保存队伍中的多个 `Sprite`
- `moveStep`：移动步长
- `team`：所属阵营
- `num`：当前存活单位数
- `buffs[BUFF_END]`：冻结、减速、防御、攻击等状态
- `score`：统计信息
- `playerType`：本地玩家、远端玩家、电脑

这意味着：

- 玩家不是控制一个单体，而是控制一个 `Snake`
- 招募英雄，本质上是给 `Snake` 头部追加一个 `Sprite`
- 死亡、Buff、武器替换很多时候是以 `Snake` 为单位传播，再作用到每个 `Sprite`

#### `Sprite`

表示 `Snake` 中的单个成员：

- 坐标 `x/y`
- 血量 `hp/totalHp`
- 当前武器 `weapon`
- 当前动画 `ani`
- 朝向 `face`、移动方向 `direction`
- `PositionBuffer posBuffer`：记录历史位置/方向，供蛇身跟随
- `lastAttack`
- `dropRate`

可以理解为：`Snake` 是逻辑编队，`Sprite` 是编队中的单兵。

#### `PositionBuffer`

内部记录一段历史位置，用来实现“蛇身按头部历史轨迹跟随”的效果。这是本项目实现贪吃蛇式队伍移动的关键数据结构。

### 3.4 战斗对象

#### `Weapon`

定义一种武器模板：

- 武器类型 `wp`
- 射程、范围、伤害、攻击间隔、子弹速度
- 出生/飞行/死亡动画
- 出生/死亡音效
- 各类 Buff 附带效果与持续时间

武器本质上是“攻击行为配置表”。

#### `Bullet`

表示投射物：

- `parent`：来源武器
- `x/y`
- `team`
- `owner`
- `rad`：飞行方向弧度
- `ani`

子弹被集中放进 `bullets` 链表，在主循环中统一移动与判定。

### 3.5 链表 ADT

`adt.h` 提供统一双向链表：

- `LinkNode`
- `LinkList`

链表被用于：

- `Snake->sprites`
- `animationsList[]`
- `bullets`

这说明项目对“动态增删对象”统一采用链表，而不是数组或 ECS。

## 4. 全局状态与控制方式

项目的一个显著特点是：核心游戏状态集中保存在全局变量中，而不是封装进一个 `GameContext` 结构。

例如：

- 地图与障碍：`map`、`hasMap`、`hasEnemy`
- 所有角色：`spriteSnake[]`
- 所有子弹：`bullets`
- 所有动画：`animationsList[]`
- 资源池：`textures[]`、`texts[]`、`effects[]`、`commonSprites[]`、`weapons[]`
- 渲染状态：`renderer`、`renderFrames`

优点：

- 写法直接，模块调用成本低
- 对课程项目/中小型 SDL 游戏足够高效

代价：

- 模块耦合比较强
- 生命周期依赖隐式，需要靠调用顺序保证正确
- 单元测试和状态隔离较困难

## 5. 执行流程

### 5.1 程序启动流程

完整入口流程可以概括为：

1. `main()` 初始化随机数。
2. `init()` 初始化 SDL、窗口、渲染器、TTF、音频、网络。
3. `loadMedia()` 加载贴图、文本、武器、公用角色模板、音频。
4. `mainUi()` 显示主菜单。
5. 用户在 UI 中选择模式后，进入 `startGame()`。
6. 程序退出时调用 `cleanup()` 回收全局资源。

### 5.2 菜单到对局的流程

`mainUi()` 中，用户选择不同选项后大致流程为：

- 单人模式：`chooseLevelUi()` -> `launchLocalGame(1)`
- 本地双人：`chooseOnLanUi()` -> `chooseLevelUi()` -> `launchLocalGame(2)`
- 局域网：`chooseOnLanUi()` -> `launchLanGame()` -> `hostGame()` / `joinGame()`
- 排行榜：`localRankListUi()`

其中 `launchLocalGame()` 会调用：

1. `startGame(localPlayerNum, 0, true)`
2. 游戏结束后展示分数界面
3. 更新本地排行榜

### 5.3 单局/多阶段流程

`startGame()` 负责一整次游戏会话，但内部可能包含多个 stage：

1. 为每个本地玩家创建 `Score`
2. `stage = 0`
3. 循环执行：
   - `initGame(...)`
   - `setLevel(gameLevel)`
   - `gameLoop()`
   - 把当前 stage 的得分累加进总分
   - `destroyGame(status)`
   - `stage++`
4. 当 `status != 0` 时退出循环

这里的含义是：

- `status == 0` 表示当前 stage 通关，但整局游戏还没结束
- 只有真正失败或最终结束时，`startGame()` 才返回给 UI

### 5.4 `initGame()` 的初始化顺序

`initGame()` 是每个 stage 的关键入口，顺序如下：

1. `randomBgm()` 随机播放背景音乐
2. 重置阶段状态、终止状态、计数器
3. `initRenderer()` 初始化渲染链表
4. `initCountDownBar()` 创建顶部倒计时条
5. 创建本地/远端玩家 `Snake`
6. 给玩家初始护盾 `shieldSnake(..., 300)`
7. `initInfo()` 初始化 UI 文字信息
8. `initRandomMap(...)` 创建随机地图
9. `clearItemMap()` 清空道具图层
10. `initEnemies(spritesSetting)` 生成怪物
11. `pushMapToRender()` 生成地图渲染层
12. `bullets = createLinkList()` 初始化子弹容器

这表明游戏世界是在“资源已存在”的前提下，为每个 stage 动态构造：

- 玩家与敌人实例
- 地图实例
- 道具实例
- 渲染动画实例

### 5.5 每帧主循环 `gameLoop()`

`gameLoop()` 是最重要的控制流程。每一帧大致按下面顺序执行：

1. 处理本地输入 `handleLocalKeypress()`
2. 若联机开启，处理网络输入 `handleLanKeypress()`
3. 更新地图状态 `updateMap()`
   - 例如陷阱/出口等地图行为
4. 遍历所有 `Snake`
   - 敌人按节流频率执行 `AiInput()`
   - `moveSnake()` 推进移动
   - `makeSnakeAttack()` 执行攻击
5. 遍历子弹链表，执行 `moveBullet()`
6. 每隔一段时间自动补充英雄与药瓶 `initItemMap(...)`
7. 更新每个 `Snake` 的动画 `updateAnimationOfSnake()`
8. 冻结状态下对动画帧做回退，制造冰冻效果
9. `makeCross()` 处理碰撞、伤害、拾取、死亡、清理
10. `render()` 绘制本帧
11. `updateBuffDuration()` 更新 Buff 持续时间
12. 清理已死亡的敌方 `Snake`
13. 判定是否进入终局倒计时或触发失败/通关

这是一个典型的单线程游戏循环：

- 输入
- 世界更新
- 战斗更新
- 碰撞结算
- 渲染
- 清理与终局判断

### 5.6 碰撞与结算逻辑

`makeCross()` / `makeSnakeCross()` 这一类函数承担了大量结算工作：

- 陷阱对角色造成伤害
- 玩家与 `Item` 相交时拾取物品
- 招募英雄后调用 `appendSpriteToSnake()` 扩充队伍
- 药瓶恢复生命
- 武器道具替换可装备角色的武器
- 角色生命 <= 0 时播放死亡音效、生成死亡动画、触发掉落
- 把死亡成员从链表中移除，并让后续身体节点前移补位

这部分是“规则密集区”，把地图、物品、战斗、渲染、音效都串起来了。

### 5.7 胜负与阶段推进

`gameLoop()` 末尾会检查：

- 是否有玩家全部死亡，如果是则进入 `GAME_OVER`
- 是否满足 `isWin()`，如果是则进入 `STAGE_CLEAR`

通过 `setTerm()` 设置一个带倒计时的终止状态，倒计时结束后返回 `status`。随后：

- `destroyGame(status)` 销毁本阶段对象与动画
- 若 `status == 0`，`startGame()` 继续下一 stage
- 否则返回 UI

因此，这个游戏不是“一局一个地图然后结束”，而是“同一局里连续推进多个 stage”。

## 6. 关键机制解读

### 6.1 队伍增长机制

`appendSpriteToSnake()` 会把新角色插入到 `Snake` 链表头部，并根据当前头部位置与朝向决定生成位置。这样招募新英雄时，队伍会像蛇一样继续延展。

移动时依靠 `PositionBuffer` 记录头部历史轨迹，身体成员沿着历史位置跟随，因此形成“队伍列队移动”的效果。

### 6.2 地图与表现分离

地图生成先写入逻辑格子：

- 哪些格子可走
- 哪些格子是陷阱
- 哪个格子是出口

之后再由 `pushMapToRender()` 根据邻接关系推导：

- 墙角
- 侧墙
- 前景墙
- 地板装饰

这使得视觉表现比纯逻辑地图更丰富，但逻辑判断仍基于规则化网格。

### 6.3 动画驱动渲染

几乎所有可见对象都被统一为 `Animation`：

- 地图块有 `Block.ani`
- 角色有 `Sprite.ani`
- 道具和特效都是单独的 `Animation`
- UI 标题和倒计时条也是 `Animation`

这是一种轻量级的“表现组件”设计。逻辑对象只要持有或绑定动画，就可以接入渲染系统。

### 6.4 配置式武器系统

武器不是通过大量分支硬编码，而是通过 `Weapon` 结构体配置：

- 攻击方式
- 射程和伤害
- 触发 Buff 的概率和持续时间
- 对应动画和音效

这让新增武器的成本主要集中在模板初始化，而不是重写主循环。

## 7. 模块依赖关系总结

可以把代码依赖粗略理解为：

- `main` 依赖 `res`、`ui`
- `ui` 依赖 `game`、`render`、`audio`、`net`、`storage`
- `game` 依赖 `map`、`render`、`player`、`sprite`、`weapon`、`bullet`、`audio`、`ai`、`net`
- `render` 依赖 `types`、`sprite`、`player`、`res`
- `map` 依赖 `types`、`render`、`res`
- `audio`、`storage`、`net` 分别提供外围能力

其中 `game.c` 是依赖汇聚点，也是最强耦合点。

## 8. 设计特点与阅读建议

### 8.1 设计特点

这个项目的工程风格有几个明显特点：

1. 过程式驱动强，核心状态通过全局变量共享。
2. 战斗、地图、道具、动画都在 `game.c` 一侧汇总调度。
3. 渲染层以 `Animation` 为统一抽象，结构清晰。
4. 数据结构比较朴素，主要依赖数组、链表、结构体。
5. 适合快速实现玩法，但后期扩展会遇到耦合增长问题。

### 8.2 建议的阅读顺序

如果后续你要继续深入代码，建议按这个顺序读：

1. `main.c`
2. `res.c`
3. `ui.c`
4. `game.c` 的 `startGame()` / `initGame()` / `gameLoop()` / `destroyGame()`
5. `player.h`、`sprite.h`、`weapon.h`、`bullet.h`
6. `map.c`
7. `render.c`
8. `ai.c` / `net.c` / `storage.c`

这样可以先建立完整控制流，再深入具体子系统。

## 9. 项目优点

从现有实现来看，这个项目有几项比较明显的优点：

### 9.1 核心玩法抽象比较统一

项目把“玩家队伍”抽象为 `Snake`，把“单个成员”抽象为 `Sprite`，再用 `PositionBuffer` 实现队伍跟随。这套抽象和游戏玩法高度贴合，不是机械套用通用实体模型，而是直接服务于“像蛇一样扩张的队伍战斗”。

这点做得很好，因为它让：

- 招募英雄
- 队伍增长
- 队形跟随
- 成员死亡补位

都能落在同一套数据结构上完成。

### 9.2 渲染层有清晰的统一抽象

`Animation` 基本统一了地图、角色、特效、UI 的表现层对象，渲染模块不需要为每类对象单独维护一套接口。这种做法虽然简单，但非常实用，尤其适合 SDL 这类偏底层框架。

它的好处是：

- 新增表现元素时接入成本低
- 动画生命周期可统一管理
- 绑定到 `Sprite` 的特效也有明确机制

对于一个中小型 2D 游戏，这比过早引入复杂引擎式架构更稳妥。

### 9.3 游戏循环主线清楚

`gameLoop()` 的顺序比较直观：输入、地图更新、AI/移动、攻击、子弹、补给刷新、动画更新、碰撞结算、渲染、Buff 更新、清理、胜负判定。即使单个函数内部逻辑较多，至少主循环层面是容易读懂和验证的。

这意味着维护者可以较快定位一类问题属于：

- 输入阶段
- 更新阶段
- 结算阶段
- 渲染阶段

对排查游戏逻辑 Bug 很有帮助。

### 9.4 配置式武器和 Buff 设计可扩展

`Weapon` 结构把伤害、范围、子弹速度、音效、动画、Buff 概率集中到一起，已经具备“数据驱动”的雏形。相比把所有武器写成大量 `if/else`，当前设计对扩展新武器明显更友好。

### 9.5 地图生成和地图表现分层合理

地图逻辑生成和地图表现拼装是分开的：先生成可走区域、陷阱、出口，再根据邻接关系推导墙面和装饰。这样做既保留了规则判断的简单性，也让画面效果不会太单调。

### 9.6 作为小型纯 C 项目，完成度较高

从工程范围看，这个项目已经覆盖了：

- 单机流程
- 局域网模式
- 音频
- UI
- 排行榜持久化
- 随机地图
- AI
- 武器与 Buff

对于纯 C + SDL 的个人项目，这种完成度是比较不错的，说明作者在“把功能真正做完”这件事上执行力很强。

## 10. 值得改进的地方

下面这些不是“代码写错了”，而是从后续维护、扩展和稳定性角度看，最值得优先改进的地方。

### 10.1 应该逐步收拢全局状态

目前项目大量依赖全局变量，例如 `spriteSnake[]`、`map[][]`、`bullets`、`animationsList[]`、`textures[]`、`renderFrames` 等。短期内这很方便，但随着功能增加，状态依赖会越来越隐蔽。

最值得优先做的改进不是“一次性重构完”，而是先引入一个 `GameContext` 或 `GameState` 结构，把游戏期状态逐步收进去，例如：

- 本局地图状态
- 实体列表
- 子弹列表
- 阶段计数器
- 终局状态

这样做的收益是：

- 初始化/销毁边界更清楚
- 联机、回放、测试时更容易复制状态
- 后续减少跨文件隐式依赖

### 10.2 `game.c` 体量过大，职责过重

目前 `game.c` 同时负责：

- 局初始化与销毁
- 掉落
- 道具刷新
- 伤害计算
- Buff 结算
- 碰撞处理
- 胜负推进

这会让文件持续膨胀，也让修改一处逻辑时更容易影响别处。比较现实的拆分方式可以是：

- `game_flow.c`：初始化、主循环、阶段推进
- `combat.c`：伤害、攻击、Buff、死亡
- `pickup.c`：道具生成、拾取、掉落
- `game_state.c`：全局计数和状态辅助函数

不一定要马上拆很多文件，但至少可以先按职责把内部函数群分层。

### 10.3 生命周期管理还可以更明确

当前代码大量手动管理：

- `Animation` 的创建与销毁
- `Snake` 与 `Sprite` 的释放
- 子弹与其绑定关系
- 死亡后从链表移除

这类代码在 C 项目里很正常，但如果没有更明确的“所有权规则”，后面很容易出现：

- 重复释放
- 悬空指针
- 某类对象漏回收

建议逐步明确每类对象的所有权，例如：

- 谁创建 `Animation`，谁负责销毁
- 绑定到 `Sprite` 的动画何时自动清理
- `Bullet.owner` 失效后哪些逻辑必须判空

文档里如果顺手补一个“对象生命周期表”，维护成本会下降很多。

### 10.4 模块边界可以更稳定一些

例如渲染、地图、游戏逻辑之间现在有较多相互感知：

- 地图初始化时直接构造动画
- 游戏逻辑直接操作渲染链表
- 角色死亡时同时清理逻辑对象和表现对象

这在小项目阶段可以接受，但如果以后要加入更多技能、更多 UI 状态或更多地图交互，边界会开始变脆。

一个实际可行的改进方向是：

- 逻辑层更多产出“状态变化”
- 渲染层更多负责把状态映射成动画

不必追求完全解耦，但可以减少“任何模块都能直接改任何渲染链表”的情况。

### 10.5 常量配置分散，缺少集中配置层

当前有很多宏和参数散落在不同头文件与源码里，例如：

- 地图刷新周期
- Buff 系数
- 掉落概率
- AI 决策频率
- 武器参数

对调平衡来说，这会让维护成本偏高。比较好的方向是把“玩法参数”集中到专门配置层，至少按类别集中：

- 关卡参数
- 玩家/怪物参数
- 掉落参数
- AI 参数

如果继续往前走，甚至可以把一部分改成可加载配置文件。

### 10.6 测试与调试支撑还比较薄

项目目前显然更偏“直接运行游戏观察效果”，这对玩法开发是有效的，但对回归验证不够友好。尤其是地图生成、伤害计算、掉落概率、Buff 持续时间这种逻辑，适合增加最小粒度的验证手段。

不一定需要完整单元测试框架，先做下面几类轻量改进就有价值：

- 给关键逻辑增加 debug 统计输出
- 给地图生成和战斗结算增加可重复种子回放
- 给核心规则函数补少量断言或自检

这样排查行为回归时会轻松很多。

### 10.7 一些实现细节可以再做健壮性整理

从代码风格看，项目已经能运行，但仍有一些典型的纯 C 项目风险点值得后续整理：

- 某些函数对外部全局前置条件依赖较强
- 部分命名和拼写不统一，例如 `DEFFENCE`、`MEDCINE` 一类历史命名
- 个别逻辑函数承担太多副作用，阅读成本偏高

这些不一定影响当前运行，但会慢慢抬高维护门槛。适合在不改行为的前提下，分批做小规模清理。

### 10.8 若继续演进，优先级建议

如果这个项目后续还要继续维护或扩展，我建议改进优先级是：

1. 先收拢游戏期状态，减少全局变量扩散。
2. 再拆分 `game.c`，把主循环、战斗、掉落、结算分开。
3. 然后梳理对象生命周期和内存所有权。
4. 最后再考虑更进一步的数据驱动和测试体系。

这个顺序的原因是：前两项能直接降低耦合，第三项能提升稳定性，第四项才是中长期演进能力。

## 11. 一句话总结

DungeonRush 的本质是一个以 `gameLoop()` 为中心、以 `Snake + Sprite + Animation` 为核心抽象、以全局状态驱动的 SDL 地牢动作游戏；它的代码结构清晰地围绕“资源初始化 -> UI 选择 -> 阶段初始化 -> 帧循环更新 -> 渲染 -> 结算/推进”这条主线展开。
